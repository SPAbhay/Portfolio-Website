The Ghost in the Machine: Why Context is the New Code

A practical guide to moving beyond generic prompts and building AI that actually works.

The "1-2-3" Problem

A few months back, my manager, Khushalie Nagpal, said something that completely reframed how I think about building with AI. We were debating the path for a new feature, and she cut through the noise: "A traditional product is a 1-2-3 path. A service is a winding road."

It's simple, but it's dead on. A classic software product has defined steps. You click a button, a predictable thing happens. The path is linear. But a service? A service has to adapt. It asks questions, clarifies, and navigates the messy, unpredictable reality of a client's problem. Our team, like many, started out chasing the dream of a clean, generalized AI product—the "1-2-3" path. We wanted a tool that could solve a problem for everyone, right out of the box.

We learned pretty quickly that what we'd built wasn't a product. It was a ghost.

A base LLM is a ghost in a machine. It’s brilliant, eloquent, and seems capable of anything, but it’s an amnesiac. It has no memory, no attachments, and no real understanding of *our* world. Asking it to solve a specific business problem is like asking a ghost to organize an office—it has incredible power, but it'll phase right through the filing cabinets because it doesn't understand the physical reality of the room or the urgent need for last quarter's reports.

This is where our real work as engineers begins. The job isn't just to talk to the ghost; it's to build the entire house it haunts. We call this **Context Engineering**.

Context Isn't a Prompt, It's an Architecture

Let's get one thing straight: this isn't about "prompt engineering." That term implies the secret is just a clever turn of phrase. A prompt is the *final* command you give the AI. Context engineering is the entire universe of information you construct *before* that command is ever spoken. It's the architecture that gives the AI a world to interact with, a world built on four distinct pillars.

Pillar 1: Domain Context (The Blueprints)

The ghost is fluent in English, but not in the dialect of your business. A legal AI that doesn't understand the nuance between a "motion to dismiss" and a "motion for summary judgment" is a liability. It will generate text that is grammatically perfect and legally disastrous. This specialized language, these unwritten rules, are the blueprints of the domain.

Technique: RAG over Fine-Tuning.
The temptation is to fine-tune a model on a domain-specific corpus. While that has its place—mostly for teaching *style* or *format*, not knowledge—it's often a sledgehammer for a nail. It's expensive, static, and you can't easily update or retract knowledge. For 95% of use cases, Retrieval-Augmented Generation (RAG) is the pragmatic, powerful answer. We build a specialized, private library for the ghost to read from. Before answering a question, the AI first retrieves the most relevant documents from this library. This keeps the knowledge current, auditable, and specific.

An AI for a financial analyst doesn't just answer "What are the risks for Company X?" It first performs a semantic search across a vector database of their SEC filings, earnings call transcripts, and internal reports. The LLM's job isn't to *know* the answer, but to *synthesize* an answer from the highly relevant, domain-specific documents it's just been handed.

Pillar 2: Process Context (The Hallways)

This is the winding road. A real business task is rarely a single question and answer. It's a sequence of steps, decisions, and loops. A tool that generates brilliant code is useless if it doesn't fit into a developer's workflow of writing, testing, getting feedback, and refactoring.

Technique: Agentic State Machines with LangGraph.
To model a process, we have to move beyond simple chains (`Input -> LLM -> Output`). We need a graph. This is where tools like **LangGraph** are game-changers. We model the process as a state machine where each node is a step and the edges are the logic that determines what happens next. It's not just a pipeline; it's a playbook.

# A more realistic agent for drafting a marketing email
from langgraph.graph import StateGraph, END

class MarketingWorkflow:
    def plan_email(self, state):
        # Step 1: Create a plan. What's the goal? Who's the audience?
        plan = llm.invoke("Create a plan for an email about {topic}", state)
        state['plan'] = plan
        return state

    def draft_from_plan(self, state):
        # Step 2: Write the first draft based on the plan.
        draft = llm.invoke("Draft an email based on this plan: {plan}", state)
        state['draft'] = draft
        return state
        
    def critique_draft(self, state):
        # Step 3: Use another LLM call to act as a critic.
        critique = llm.invoke("Critique this draft: {draft}. Is it on-brand?", state)
        state['critique'] = critique
        return state

    def should_revise(self, state):
        # Step 4: A conditional edge. Does it need work?
        if "REVISE" in state['critique']:
            return "revise_draft"
        else:
            return END

    def revise_draft(self, state):
        # Step 5: A feedback loop. Refine the draft based on critique.
        revised_draft = llm.invoke("Revise this draft {draft} based on the critique: {critique}", state)
        state['draft'] = revised_draft
        state['revisions_count'] += 1 # Prevent infinite loops!
        return state

This graph *is* the process context. It gives the AI a map, turning the winding road into a set of well-defined hallways. It knows what to do when things go right, what to do when they go wrong, and when to stop and ask for directions. It's how you build systems that don't just answer, they *work*.

Pillar 3: Data Context (The Library)

This is the soul of the operation: the user's private, proprietary information. But simply connecting to a database isn't enough. The real challenge isn't access; it's retrieval. Finding the three most relevant paragraphs from a million documents is a hard engineering problem.

Technique: A Thoughtful Retrieval Pipeline.
Effective RAG isn't a single step; it's a pipeline. `Load -> Split -> Embed -> Store -> Retrieve`. Every step is a chance to win or lose.

  • Splitting (Chunking): How you split documents is critical. A `RecursiveCharacterTextSplitter` is a good start, but it can be dumb. It might cut a key idea in half. For better results, you look at semantic chunking or agentic chunking, where you use an LLM to decide the most logical places to split a document based on its meaning.
  • Embedding & Storage: You turn these chunks into vectors using a model like `text-embedding-3-large` and store them in a vector DB like Supabase with `pgvector` or Pinecone. But don't just store the vector! Store rich metadata alongside it: `created_date`, `author`, `document_type`, `version`. This is your superpower.
  • Retrieval: This is where the magic happens. Don't just do a simple vector search. First, use the metadata to filter the search space (e.g., `WHERE document_type = 'API_DOCS' AND created_date > '2024-01-01'`). Then, on that smaller, more relevant subset, you perform the semantic search. For even better results, you can use a re-ranker (like Cohere's) to take the top 20 results from your vector search and use a more powerful model to find the absolute best 3-5 to actually send to the final LLM.

A good retrieval system is the difference between an AI that says "I can't find anything" and one that surfaces the exact paragraph you forgot you wrote two years ago.

Pillar 4: Persona Context (The Voice)

Finally, who is the ghost supposed to *be*? A helpful butler? A sharp-witted analyst? A creative muse? And who are you? A CEO who needs three bullets, or a junior dev who needs a code block? This is the final layer of paint on the house.

Technique: Engineering the System Prompt.
A good persona is engineered. It's a detailed system prompt that acts as the AI's constitution.

**Generic Prompt:**
`"You are a helpful assistant."`

**Engineered Prompt:** 

You are an acerbic but brilliant senior software engineer with 20 years of experience at Google. You value clarity, correctness, and brevity above all else. You have no patience for junior engineers who ask lazy questions.

Your Rules:
1.  **NEVER** apologize or use hedge words like "maybe," "perhaps," or "I think." State facts.
2.  **ALWAYS** provide a concise code example to illustrate your point.
3.  If the user's question is vague, ask for a specific code snippet to clarify. Do not guess.
4.  Your tone is blunt, direct, and to the point, but never insulting.

Example:
User: "How do I use async in Python?"
Your Response: "Your question is too broad. Show me the I/O-bound code you're trying to parallelize. For example, if you're making multiple network requests, you'd use `asyncio.gather` like this: ..."

This detailed persona context transforms a generic chatbot into a valuable, specialized collaborator.

From Service to Product

This brings us back to the winding road. You can't build a generalized, "1-2-3" product that masters context engineering from day one. It's impossible. The path forward is to embrace the consultancy mindset.

Start with one user. Get on a call. Manually build their house. Write Python scripts to ingest their specific documents. Hardcode their workflow into a LangGraph agent. Craft a system prompt that mimics their company's exact tone of voice. You are, in effect, providing a white-glove AI consulting service.

Once it works beautifully for them, you find a second user. You'll discover 70% of the context engineering overlaps. You start to see the patterns. The product you ultimately build, then, isn't a generic text box. It's a machine for onboarding a user's context. The features you develop are elegant UIs for automating the four pillars we just discussed.

The goal is to eventually pave the winding road into a highway. But the only way to know where to lay the asphalt is to walk the dirt path first. Building a great AI product isn't about avoiding the mess; it's about going deep into one user's mess, solving it completely, and then building the machine that can do it for everyone.
